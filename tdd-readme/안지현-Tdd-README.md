## TDD 학습 과정 정리
Hub 서비스에서 허브 생성을 위한 Hub 도메인, Hub Repository, Hub Service에 대해 TDD를 적용해보았습니다.
테스트 코드 클래스는 다음과 같습니다.
HubServiceTest - Service
HubRepositoryTest - Repository

## 각 Step별 학습 내용과 느낀 점
Step1.
허브 도메인에 대한 로직을 검증하기 위해 Service 단위 테스트를 먼저 작성했습니다.
허브 생성 시 필수 값 검증, 존재하지 않는 허브 조회 시 예외 발생 등
예외 상황을 미리 정의하고 테스트를 통해 예상된 동작을 검증했습니다.
테스트를 작성하면서, 허브의 상태(hubStatus)나 논리삭제(deletedAt) 처리에 대한
비즈니스 제약사항을 명확히 정의할 수 있었습니다.

Step2.
허브 Repository 테스트에서는 @DataJpaTest를 이용해 실제 DB 환경에서 CRUD 동작을 검증했습니다.
특히, 논리삭제된 데이터가 조회되지 않도록 필터링되는지를 테스트를 통해 확인했습니다.
테스트를 통해 JPA 매핑 오류나 누락된 칼럼 정의를 빠르게 발견하고 수정할 수 있었습니다.

Step3.
Controller 계층은 아직 완성되지 않아 테스트는 주석처리했지만,
Service와 Repository 테스트만으로도 기능의 핵심 로직은 충분히 검증되었습니다.
테스트 코드를 작성한 후 구현을 맞추는 방식(TDD)으로 진행하다 보니
단순히 “코드가 돌아가는가”보다 “올바른 동작인가”를 더 깊게 고민하게 되었습니다.

## 회고
Service와 Repository 단위에서의 TDD는 비교적 간단했지만,
실제 비즈니스 로직을 깊이 있게 이해해야 테스트를 설계할 수 있음을 느꼈습니다.
테스트 코드를 먼저 작성해두니 구현 중에도 “어떤 경우를 놓치고 있는가”를 자연스럽게 점검할 수 있었습니다.
Controller 테스트를 완성하면 전체 흐름 단위의 E2E 테스트까지 확장해볼 계획입니다.
